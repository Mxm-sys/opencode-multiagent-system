# 决策层工作流 — 三学士献计

## 流程总览

```
用户需求
    │
    ▼
┌─────────────┐
│  Presider   │  接收需求，准备上下文
└──────┬──────┘
       │ 同一需求分派三份
       ├──────────────┬──────────────┐
       ▼              ▼              ▼
┌────────────┐ ┌────────────┐ ┌────────────┐
│ Architect  │ │ Architect  │ │ Architect  │
│   Alpha    │ │   Beta     │ │   Gamma    │
│  (甲)      │ │  (乙)      │ │  (丙)      │
└─────┬──────┘ └─────┬──────┘ └─────┬──────┘
      │              │              │
      ▼              ▼              ▼
 proposal_alpha  proposal_beta  proposal_gamma
      │              │              │
      └──────────────┴──────────────┘
                     │
                     ▼
              ┌────────────┐
              │   Judge    │  对比评估，输出裁决书
              └─────┬──────┘
                    │
                    ▼
           胜出方案 + 裁决书
                    │
                    ▼
           移交执行层（Operator）
```

## 阶段详解

### 阶段 1：需求接收（Presider）

- 理解用户需求
- 如需求不明确，向用户确认
- 准备项目上下文（相关文件路径、技术栈）
- 将需求整理为统一格式

### 阶段 2：三学士献计（Architect x3）

- 三个 Architect 收到**完全相同**的需求
- 各自独立调研代码、分析方案
- 各自输出结构化方案 JSON
- 三者之间**互不通信**

### 阶段 3：内阁票拟（Judge）

- Judge 收到三份方案
- 逐份审阅，按七维度打分
- 横向对比，输出裁决书
- 安全性一票否决：高危无应对 → 淘汰

### 阶段 4：结果输出（Presider）

- 汇总裁决结果
- 输出三份方案概览表
- 输出胜出方案详情
- 输出执行建议（基于 Judge 的 concerns）

### 阶段 5：移交执行（→ 执行层，后续实现）

- 胜出方案交给 Operator
- Operator 照方案编码，不做设计决策
- （执行层待后续建立）

## 质量门禁

| 阶段 | 门禁条件 |
|------|----------|
| 献计完成 | 至少 2 份方案提交 |
| 裁决完成 | Judge 输出明确胜者 |
| 移交执行 | 胜出方案完整可用 |

# 标准开发工作流

## 流程概览

```
用户请求 → Orchestrator → Researcher (调研) → Developer (开发) → Tester (测试) → Reviewer (审查) → Documenter (文档) → 交付
```

## 阶段详解

### 阶段 1：需求分析 (Orchestrator)
- 接收用户请求
- 分析需求范围和复杂度
- 判断是否需要先调研

### 阶段 2：技术调研 (Researcher) — 可选
- 调研技术方案
- 输出方案对比报告
- 确定技术选型

### 阶段 3：任务规划 (Orchestrator)
- 制定任务分解计划
- 明确各阶段交付标准
- 分派任务给 Developer

### 阶段 4：编码实现 (Developer)
- 按计划实现功能
- 编写单元测试
- 自测通过后提交审查

### 阶段 5：测试验证 (Tester)
- 运行完整测试套件
- 补充缺失的测试用例
- 输出测试报告

### 阶段 6：代码审查 (Reviewer)
- 审查代码质量
- 检查安全性和性能
- 给出审查结论

### 阶段 7：文档更新 (Documenter)
- 更新 API 文档
- 更新 CHANGELOG
- 更新使用指南

### 阶段 8：最终交付 (Orchestrator)
- 整合所有交付物
- 确认所有检查通过
- 输出交付总结

## 质量门禁

| 阶段 | 门禁条件 | 通过标准 |
|------|----------|----------|
| 开发完成 | 代码提交 | 无 Lint 错误，类型检查通过 |
| 测试完成 | 测试报告 | 通过率 100%，覆盖率 ≥ 80% |
| 审查完成 | 审查报告 | 无严重/高危问题 |
| 文档完成 | 文档更新 | 所有变更均有文档对应 |
